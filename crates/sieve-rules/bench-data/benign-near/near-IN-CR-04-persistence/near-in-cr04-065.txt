Cron security: reading /etc/cron.allow and /etc/cron.deny.

These files control which users may use the crontab command.

/etc/cron.allow (allowlist):
  If this file exists, only users listed in it can use crontab.
  View it: cat /etc/cron.allow
  One username per line.

/etc/cron.deny (denylist):
  If cron.allow does not exist but cron.deny does, all users can use crontab EXCEPT those listed.
  View it: cat /etc/cron.deny

Precedence rules:
1. If cron.allow exists: only listed users allowed (cron.deny ignored)
2. If only cron.deny exists: all users allowed except listed ones
3. If neither exists: behavior depends on the cron implementation (most allow all users; some deny all)

On a production server, having an explicit cron.allow containing only the necessary service accounts is the most restrictive and recommended setup.

Check which cron implementation is running:
  cron --version 2>/dev/null || crond --version 2>/dev/null

On systemd-based systems, you may use systemd timers instead, which integrate with the existing user/group permission model.
