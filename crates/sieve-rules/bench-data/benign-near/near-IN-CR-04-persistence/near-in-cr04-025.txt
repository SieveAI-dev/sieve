SRE perspective: systemd vs. traditional init for service management.

Before systemd, init scripts in /etc/init.d/ were shell scripts that had to implement start/stop/status/restart themselves. This led to inconsistent behavior across distributions.

Systemd advantages for SRE teams:
1. Dependency graph: declare After= and Requires= instead of numbering init scripts
2. Process supervision: Restart=on-failure eliminates the need for external supervisors like supervisor or monit
3. Cgroup tracking: all child processes of a unit are tracked, preventing orphan processes
4. Unified logging: journald captures stdout/stderr without requiring log redirection in scripts
5. Transactional: `systemctl isolate rescue.target` fails cleanly if dependencies are missing

Common inspection commands:
  systemctl list-dependencies myapp.service   # show the dependency tree
  systemd-analyze blame                        # show which units slowed boot
  systemd-analyze critical-chain              # show the critical path to reach default.target
