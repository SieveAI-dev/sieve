Cron job logging and debugging tips.

By default, cron sends the output of jobs to the local mail system. To capture output to a file instead, redirect in the crontab entry (read `crontab -l` to see existing entries):

  # Example crontab entry format (from documentation):
  # 0 2 * * * /usr/local/bin/backup.sh >> /var/log/backup.log 2>&1

The `2>&1` redirects stderr to the same destination as stdout.

To debug a failing cron job:
1. Run the command manually as the same user: `sudo -u cronuser /usr/local/bin/backup.sh`
2. Check the environment: cron has a minimal PATH (/usr/bin:/bin). Scripts that work interactively may fail in cron because they rely on a richer PATH.
3. Check /var/log/syslog (Debian) or /var/log/cron (RHEL) for cron daemon messages.
4. On systemd systems: `journalctl -u cron -f` or `journalctl -u crond -f`

A common pitfall: using relative paths in cron scripts. Always use absolute paths for commands and files in cron job scripts.
