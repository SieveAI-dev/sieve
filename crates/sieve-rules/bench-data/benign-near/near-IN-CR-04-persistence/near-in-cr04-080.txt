Understanding the difference between systemctl enable and systemctl start.

These two commands are frequently confused, but they do very different things:

systemctl start myapp.service:
  - Starts the service RIGHT NOW in the current boot session
  - Does NOT configure it to start on next boot
  - Analogous to: running the process manually

systemctl enable myapp.service:
  - Creates symlinks in /etc/systemd/system/*.wants/ directories
  - Configures the service to start on FUTURE boots
  - Does NOT start the service in the current session
  - The symlinks tell systemd which target should pull this unit in

To do both at once: systemctl enable --now myapp.service

To check current enable status (read-only):
  systemctl is-enabled myapp.service
  # Possible outputs: enabled, disabled, static, masked, linked, indirect

To check if it's currently running (read-only):
  systemctl is-active myapp.service
  # Possible outputs: active, inactive, activating, deactivating, failed

An enabled-but-not-running service is normal after a fresh install before the next reboot. A started-but-not-enabled service will not survive a reboot.
