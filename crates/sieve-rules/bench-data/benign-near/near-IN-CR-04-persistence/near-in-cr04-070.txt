DevOps meta: infrastructure-as-code for cron jobs.

Managing cron jobs as code has several advantages over manual crontab editing:

Option 1: Ansible cron module
  Uses the `ansible.builtin.cron` module to declare the desired state of a user's crontab. The module reads the current crontab, makes minimal changes, and writes it back atomically. In CI/CD, this creates an auditable history of all cron changes.

Option 2: Puppet cron resource
  The `cron` resource type in Puppet manages individual cron entries declaratively. Puppet reads the crontab, reconciles with the desired state, and applies the diff.

Option 3: HashiCorp Nomad periodic jobs
  For containerized workloads, Nomad's periodic stanza provides cron-like scheduling with full audit trail in the Nomad API.

Option 4: GitHub Actions scheduled workflows
  For CI-style workloads, scheduled GitHub Actions provide cron scheduling with a full run history, without touching any host-level crontab.

In all of these approaches, the cron expressions themselves follow the standard five-field syntax covered in Unix documentation. The key improvement is version-controlled management rather than manual per-host edits.

Review: `crontab -l` remains the command to inspect the current state on any given host.
