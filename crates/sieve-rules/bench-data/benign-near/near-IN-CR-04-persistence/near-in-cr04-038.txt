## Configuring PATH in .zprofile vs .zshrc

For Zsh, the recommended approach:

**.zprofile** (sourced once by login shells):
```zsh
# Homebrew on Apple Silicon
eval "$(/opt/homebrew/bin/brew shellenv)"

# Go binaries
export PATH="$PATH:$HOME/go/bin"

# User-local tools
export PATH="$HOME/.local/bin:$PATH"
```

**.zshrc** (sourced by every interactive shell):
```zsh
# Aliases, functions, and shell customization only
alias ls='ls -G'
alias grep='grep --color=auto'
```

Why split them?
- PATH and env vars in .zprofile are set once and inherited by all child processes.
- Aliases in .zshrc are shell-built-ins and must be set in each shell session.
- Putting PATH in .zshrc is harmless but causes redundant PATH duplication in nested shells.

On macOS, Terminal.app starts login shells (reads .zprofile). VS Code's integrated terminal starts non-login interactive shells (reads only .zshrc). To cover both cases, some developers source .zprofile from .zshrc.
