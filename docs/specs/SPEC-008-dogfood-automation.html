1.概述与核心结论

结论先行：Sieve 的 dogfood 验证由一份需要真 ANTHROPIC_API_KEY + 真网络 + 真 macOS GUI 才能逐项勾选的手动 checklist，重构为全程 hermetic（本地 mock 上游 + headless CLI）、一条命令 scripts/dogfood.sh 跑通、CI 零 secret 复现的自动化端到端测试。手动 checklist 不依赖真模型/真网络的能力块绝大多数已迁入自动化；仅依赖外部真 API 与 GUI 可视层的部分保留手动。自动化覆盖的范围内，端到端内容回环断言能抓出仅靠单测无法暴露的真实问题（见第 6 节）。

两条范围决策

① headless 客户端替代 GUI

决策客户端不再依赖真 macOS GUI App。daemon 已经是 client-agnostic（协议中性化）：用 sieve decisions headless CLI / 直连 IPC socket 的 mock GUI 当决策客户端，跑通 watch / approve / block / warn 全链路。GUI 仓只保留跨仓 17-method fixture 一致性测试，不碰 XCUITest 可视层。

② 全程 hermetic mock 上游

本地一键 和 CI 无 secret 都要能跑 → 唯一可行解是全程 mock 上游。mock 同时覆盖 Anthropic（SSE + 非流式 JSON）和 OpenAI（SSE + JSON）四类 content-type 组合（硬约束 #16）；daemon 经 tls_verify_upstream=false 走 plain HTTP 连 mock。出站拦截（426 / 脱敏）由 daemon 自身完成，与上游无关。

harness 形态

  • 新 dev crate crates/sieve-testing：把分散在 sieve-cli/tests/* 的 mock 上游 / daemon 进程管理 / 原始 HTTP client / CLI 驱动 helper 抽取为可复用库。模块：paths（端口 / 二进制 / 规则路径）· upstream（mock 上游 + Anthropic/OpenAI 响应生成器）· daemon（DaemonGuard + DaemonConfig + wait_for_ipc）· http（含 decode_chunked）· cli（run_sieve_cli shell out 真 CLI）。
  • cargo 集成测试为主：sieve-cli/tests/dogfood_e2e.rs（Phase A–D，11 个测试）+ sieve-updater/tests/updater_e2e.rs（§14 闭环，8 个测试）。
  • Python smoke 为辅：scripts/smoke_test.py --mock-only（黑盒全量通过）。
  • 一键入口：scripts/dogfood.sh 串起构建 → cargo e2e → smoke → updater 闭环，退出码 0/非 0 供 CI 直接判定。

2.架构总览

测试驱动器（cargo test / smoke）在同一进程内起 mock 上游，spawn 真实 sieve daemon 进程指向它，再以 headless CLI / mock GUI 当决策客户端，最后对 HTTP 响应、上游收到的 body、audit.db 做断言。整条链路无真 key、无真网络、无真 GUI。

已自动化（真 daemon / 真断言） hermetic mock（替身） 保留手动

写测试时探针核实的 daemon 真实行为 dogfood_e2e 不是凭空写的——落地时用探针逐项核实并在文件头记录了几条非显然行为：① no_client_policy 只从 CLI flag 读、从不读 sieve.toml（且 Config 有 deny_unknown_fields），故 Phase C 改用直接 spawn 二进制 + flag；② wait_for_ipc 探测连接会污染 connected_clients，测「无 client」路径时刻意不调它。

4.hermetic mock 基建

4.1 mock 上游（Anthropic + OpenAI 全协议）

sieve-testing::upstream 提供两类 mock：spawn_mock_upstream（Full<Bytes> body，模拟非流式 JSON）和 spawn_mock_streaming_upstream（StreamBody chunked，无 content-length，模拟 SSE）。responses 子模块按真实协议骨架生成 Anthropic（message_start…message_stop、tool_use + input_json_delta）和 OpenAI（chat.completion.chunk + [DONE]）的 SSE/JSON，外加 auth_error_401。这覆盖硬约束 #16 要求的 4 类 content-type 组合。

4.2 SIEVE_UPDATE_ALLOW_HTTP 接缝（release 编译期消除）

updater 的 download.rs / manifest.rs 原本双 https_only()，使本地 plain-HTTP mock 无法联调。手动 checklist §14.3 的旧方案是 caddy + mkcert 起 https 反代——复杂、依赖外部工具、CI 不友好。新方案改在 tls.rs 加一个编译期接缝：仅 #[cfg(debug_assertions)] 下读 SIEVE_UPDATE_ALLOW_HTTP 放宽为 https_or_http()；release/GA 构建里 allow_plain_http() 恒返回 false 且整个降级分支被编译期消除——生产二进制根本不存在降级到明文的代码路径。这保住了「唯一允许出站是 TLS manifest/CDN」的安全不变量，同时让测试零 TLS 依赖。

4.3 SIEVE_CACHE_DIR 隔离

cache_dir.rs 原本硬用 $HOME/Library/Caches/sieve，测试会污染真实用户缓存。新增 CACHE_DIR_ENV：置非空值时逐字用作缓存目录（verbatim），绕过平台检测。install-id 与 updater runner 都经 cache_dir() 自动获益。override 优先级用纯函数 resolve_cache_dir_with_override 单测，避免改进程全局 env 在并行 runner 下竞态。这只控制文件落点，签名校验等安全行为不受影响。

5.updater 闭环时序（§14 自动化）

updater_e2e.rs 用本地 plain-HTTP mock（复用 spawn_mock_upstream）跑通完整规则热更新闭环，跨平台 hermetic（非 macOS-only）。mock manifest 的 rules.url 自引用本机 Host，下载回环到同一 mock。

断言强度：闭环测试不止查「文件存在」，而是断言 <version>.json 内容 == 解压后的原始规则 JSON、current.json 解析到同一内容、latest_version.json 记录版本号——正是这个端到端内容回环断言抓出了下文第 6 节的 ZSTD_MAGIC bug。遥测维度：telemetry 开启时 uid 出现在 manifest query，关闭（uid=None）时缺失。

6.价值印证：自动化首跑抓出的 2 个真 bug

自动化的价值不是「让绿灯更绿」，而是把没法自动跑 = 必然漂移的验证拉回每次都跑。首次落地即抓出 2 个真问题——其一是错误断言，其二会让整条生产功能失效。

BUG-1 OUT-01 出站断言滞后于 disposition 改动（426 → auto_redact）

给 smoke_test.py 加 --mock-only 时，test_outbound_block_fake_key 断言「OUT-01 fake key → HTTP 426 + sieve_blocked」实跑得到 401。OUT-01 的 disposition 早已从 block 改为 auto_redact（二维处置矩阵）——daemon 现在脱敏后转发而非 426 拦截。Rust 集成测试当时同步改对了，Python smoke 没改，断言变成错的。

为什么旧测试没抓到：smoke 默认要真 ANTHROPIC_API_KEY + 真网络才跑全套，依赖外部 key 的测试无法在常规 CI 里跑，断言滞后于代码变更不会被及时暴露。修复：smoke 断言改为验「转发到上游的 body 已脱敏」（mock 捕获 body 比对，含 REDACTED 占位符）而非验状态码，--mock-only 全量通过。

BUG-2 sieve-updater ZSTD_MAGIC 字节序反置 —— 真实规则包永不解压（生产级）

install.rs 里 ZSTD_MAGIC = [0xFD,0x2F,0xB5,0x28] 字节序反了。zstd 帧 magic 是 0xFD2FB528，磁盘小端存储为 28 B5 2F FD（RFC 8878 §3.1.1）。于是 data[..4] == ZSTD_MAGIC 对任何真 zstd 流永远为假 → decompress_zstd 永远走「当原始字节」fallback → 规则包被原样（压缩态）写盘 → sieve-rules 加载必失败。整个规则热更新功能在生产中是坏的。

为什么旧测试没抓到：现有 4 个 zstd 单测全部假阳性掩盖——要么用明文走 fallback、要么用同一个错误常量当 payload 前缀（与 bug 自洽）、要么只断言「文件存在 / 版本号」从不断言解压后内容。「测试通过」≠「功能正确」。唯有 updater_e2e.rs 用真实 zstd payload 做 encode→install→read 端到端内容回环断言才能抓出。修复：常量改 [0x28,0xB5,0x2F,0xFD] + happy_path 补解压内容断言；updater 单测与 e2e 全绿。

7.运行方式

7.1 本地一键

# 完全 hermetic：无需真 ANTHROPIC_API_KEY、无需真网络、无需真 GUI
cd ~/src/sieve-suite/sieve
scripts/dogfood.sh            # 全套：构建 → cargo e2e → smoke → updater 闭环
scripts/dogfood.sh --fast     # 复用已存在二进制，跳过 release 构建

dogfood.sh 自带 PATH 兜底（cargo 装在 ~/.cargo/bin 但 shim 未必进 PATH），CI 无 tty 时降级为纯文本输出，退出码 0/非 0 直接供 CI 判定。它只跑 -p sieve-testing + dogfood_e2e + smoke + updater_e2e，不触全 workspace 高并发下 outbound_block.rs::r11 的存量 flake。

7.2 CI 无 secret 跑法

因全程 hermetic，CI 不需要任何 secret。规划矩阵：

# ubuntu 通用 job（无 macOS-only 依赖，updater_e2e 跨平台 hermetic）
cargo build --release -p sieve-cli --locked
cargo test -p sieve-testing --locked
cargo test -p sieve-cli --test dogfood_e2e --locked
cargo test -p sieve-updater --test updater_e2e --locked
python3 scripts/smoke_test.py --mock-only

# macOS 全套 job + 跨仓 fixture block PR —— 零 secret

注：基线 cargo 是 ~/.cargo/bin 的 1.88，非 PATH 默认；脚本自带兜底。CI 用 nextest retries 治理高并发 flake。

8.保留手动的部分（及理由）

保留手动 真 API 出/入站规则触发

真 Claude Code / Codex 发起的 OUT-* / IN-CR-* 真实流量触发不进 e2e。理由：Critical 拦截误报率必须维持在极低阈值这条质量承诺由离线 vectorscan fuzz + deterministic 样本集保证，确定性远高于真模型的随机输出；e2e 用 mock 固定 payload 验「检测逻辑接线正确」即可，FP 率不该靠端到端跑真模型来证明。

保留手动 GUI 可视层

菜单栏 5 状态 / Toast / Settings 6 Tab / Onboarding 权限引导仍由人工 dogfood（一轮 2–3h）。理由：决策逻辑已被 headless 客户端完整覆盖（Phase C/C2），剩下的是纯视觉/交互层，自动化它需要 XCUITest 且收益低；范围决策 ① 明确 GUI 仓只保跨仓 fixture 一致性测试，不碰 XCUITest。
